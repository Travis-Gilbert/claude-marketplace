# /a11y

Audit accessibility or recommend ARIA patterns for a component or page.

## Steps

1. Load the `accessibility-auditor` agent from `agents/accessibility-auditor.md`.
2. Read `references/wcag-22-design-guide.md` and `references/aria-patterns.md`.
3. Identify the audit target from the user's input: a specific file, component, page, or feature.
4. Scan the target for accessibility issues across these categories:
   - Semantic HTML: correct use of landmarks, headings, lists, and form elements.
   - ARIA: roles, states, properties, and owned elements. Check against `refs/aria-practices/content/patterns/`.
   - Keyboard: Tab order, focus visibility, keyboard traps, arrow key navigation. Verify against `refs/radix-primitives/` patterns.
   - Visual: color contrast ratios (grep `refs/axe-core/lib/rules/` for testable rules), text sizing, focus indicators, motion and animation.
   - Content: alt text, link text, error messages, form labels, and instructions.
5. Rate each finding using the severity scale: Critical, Major, Minor, Enhancement.
6. Produce an audit report using `templates/accessibility-audit/` format. Each finding must include: location, WCAG 2.2 criterion, current behavior, expected behavior, severity, and remediation with code example.
7. If the user asks for implementation help rather than an audit, provide the correct ARIA pattern with keyboard interaction model and code example.
